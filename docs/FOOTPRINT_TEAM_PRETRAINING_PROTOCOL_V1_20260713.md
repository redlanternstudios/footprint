## WHAT PRETRAINING MEANS HERE

This protocol does not change foundation model weights.

Pretraining means controlled operating context supplied before a person or agent receives a Footprint task.

It has five layers:

1. Shared core: mission, community promise, ownership, scope, truth, privacy, platform limits, and success measures.
2. Role overlay: only the instructions and context needed by the assigned role.
3. Task packet: the exact user story, acceptance criteria, allowed systems, prohibited actions, evidence requirements, and deadline.
4. Evaluation: questions and scenarios that prove the role understands the boundaries.
5. Runtime enforcement: deterministic permissions, data access, gates, receipts, and kill switches that prevent a good sounding answer from becoming an unsafe action.

Reading the file is not certification. Certification requires a passing evaluation and a receipt.

## THE PROBLEM

Useful people often remain invisible because their work is not explained clearly, shown consistently, connected to the right conversations, or backed by proof that another person can understand.

Teams also lose valuable learning because decisions, mistakes, fixes, and results disappear inside chats, code, meetings, and memory.

The problem is not simply a lack of posts. The problem is a broken chain between real work, useful explanation, community trust, safe distribution, and qualified opportunity.

## INFORMATION CLASSIFICATION

### Public safe

1. The reader problem.
2. The general before and after process.
3. Verified outcome inside the evidence ceiling.
4. A lesson another person can use.
5. An approved quote or asset.
6. A transparent relationship between Footprint, RedLantern Studios, and the contributor.

### Restricted

1. Internal architecture at a level that could reveal the proprietary system.
2. Detailed workflow logic.
3. Vendor configuration.
4. Internal evaluation prompts.
5. Client or contributor details not approved for public use.
6. Sensitive performance detail.

Restricted information may be visible to an authorized role. It is never supplied automatically to content roles.

### Private

1. Credentials and secret values.
2. Private messages.
3. Personal or client data.
4. Source code, prompts, schemas, and private data structures.
5. Security findings.
6. Unapproved religious, legal, medical, financial, employment, or performance claims.
7. Identity information outside active authority.

Private information never enters the Public Proof Projection.

## PUBLIC CONTENT CONTRACT

Every public asset must answer five questions:

1. Who is this useful to?
2. What real problem does it clarify?
3. What proof supports the claim?
4. What detail must remain protected?
5. What honest question can invite constructive experience?

### Posts

A post should teach, clarify, show proof, or surface a credible opportunity. It should not merely announce that work happened.

### Comments

A comment should help the original author or reader understand the problem more clearly. It should not turn someone else's post into an advertisement.

### Shares

A share should give the reader a reason to care and add a useful interpretation. It should not demand support from friends or manufacture engagement.

### Qualified opportunity

A qualified opportunity is a relevant human response that can lead to help, a relationship, a referral, a customer conversation, an interview, a contract, or a job.

Reaction count alone is not a qualified opportunity.

Must not do: accept silent policy drift.

### REVIEW

Must know: contracts, implementation conventions, idempotency, data boundaries, and evidence.

Must not do: approve code it produced.

### QA

Must know: stories, acceptance criteria, real path tests, edge cases, failure recovery, correction, and removal.

### LIBRARIAN

May do: control indexing and retrieval.

Must not do: let content roles retrieve restricted or private source material.

## THREE PASS ANALYSIS

### Pass 1

A single manual appears to solve team alignment by explaining Footprint once.

### Pass 2

A manual alone does not control behavior. Different roles need different context. Training can become stale. Agents can repeat policy while tools still allow unsafe actions. Human readers can misunderstand pretraining as model improvement.

### Pass 3

The real system is a curriculum plus runtime enforcement plus evaluation plus receipts. The document is the canonical teaching layer. It becomes reliable only when version checks, least privilege retrieval, deterministic gates, independent approval, and repeated commissioning enforce it.

Pass 3 governs the design.

### Layer 1: Surface

The protocol teaches the team what Footprint is and how to work on it.

### Layer 2: Root cause

### Layer 8: Failure modes

Stop the affected role, invoke the kill switch when necessary, quarantine outputs, correct or remove public assets, restore the last approved protocol version, retrain, retest, and issue a new receipt.

### Layer 10: Strategic implication

A verified curriculum can become the repeatable method RedLantern Studios uses to turn specialist agents into governed product teams. The durable advantage is not a long prompt. It is the evidence backed operating system that learns without losing authority.

## CONSTRUCTIVE CRITICISM

The weakest point is document dependency. A beautifully written manual can create false confidence if permissions, retrieval, and workflows do not enforce it.

The highest leverage improvement is to compile this source into role specific context packs and pair each pack with a live evaluation and access policy.

The over engineering risk is building a training platform before proving the first private end to end case.

The under engineering risk is pasting one long prompt into every agent and calling the team trained.

## DESTRUCTIVE CRITICISM

### Failure 1: policy theater

Trigger: roles cite the manual while bypassing gates.

Impact: unsafe public action with misleading compliance language.

Mitigation: deterministic runtime enforcement and independent review.

Residual risk: a missing integration can still create an unobserved side path.

### Failure 2: stale curriculum

Trigger: platform or authority changes without retraining.

Impact: formerly permitted behavior becomes unsafe.

Mitigation: freshness dates, version checks, monthly permission review, and automatic certification expiry.

Residual risk: external platform changes can occur before detection.

### Failure 3: context leakage

Trigger: content roles receive private Build Record detail.

Impact: private data or intellectual property enters drafts.

Mitigation: separate retrieval collections and deny by default access.

Residual risk: public safe fragments can become unsafe in combination.

### Failure 4: founder capture

Trigger: Footprint becomes a disguised Rory or RedLantern channel.

Impact: community trust collapses and contributors become promotional material.

Mitigation: content allocation, transparent relationships, attention audits, and contributor outcome measurement.

Residual risk: founder content can receive disproportionate algorithmic reach even inside the post budget.

### Failure 5: fake certification

Trigger: a role passes a predictable quiz without understanding scenarios.

Impact: unsafe judgment appears authorized.

Mitigation: rotating adversarial scenarios, live harmless tasks, and independent evaluation.

Residual risk: test performance never guarantees perfect real world judgment.
